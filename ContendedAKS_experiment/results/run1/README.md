# Run results — methodology & interpretation

This folder contains the per-task CSVs produced by one execution of the
contended-AKS experiment, plus the aggregated `summary.txt` and `cdf.png`
generated by [`analyze.py`](../../analyze.py).

## 1. What was measured

A single Pod with two containers (`rt-a`, `rt-b`) was scheduled on the same
AKS node (`Standard_D2ps_v6`, 2 vCPUs, ARM64). Each container spawned 3
periodic threads, for a total of **6 CPU-bound periodic tasks contending on
2 vCPUs**.

| Task | Period T | Budgeted runtime C | U = C/T |
|------|---------:|-------------------:|--------:|
| 0    | 20 ms    | 5 ms               | 0.25    |
| 1    | 50 ms    | 15 ms              | 0.30    |
| 2    | 100 ms   | 25 ms              | 0.25    |

Per container U = 0.80; with two containers the offered load is **U = 1.60
on 2 cores**, i.e. ~80 % utilization with 20 % headroom. The task set is
trivially schedulable under a real-time policy (e.g. SCHED_DEADLINE), so
any deadline miss observed here is attributable to the scheduler in vanilla
AKS, not to overload.

## 2. How each invocation is timed

For every invocation `i` of every task, the workload records:

| Column | Clock | Meaning |
|---|---|---|
| `release_ns` | `CLOCK_MONOTONIC` | Absolute scheduled release time of invocation `i` (= start of the period). The thread was sleeping on `clock_nanosleep(TIMER_ABSTIME)` until this instant. |
| `start_ns`   | `CLOCK_MONOTONIC` | Wall-clock instant at which the thread actually returned from sleep and entered the busy loop. `start_ns − release_ns` is the **release jitter**. |
| `finish_ns`  | `CLOCK_MONOTONIC` | Wall-clock instant at which the busy loop completed. |
| `response_us`| derived           | `(finish_ns − release_ns) / 1000` — the **response time**, including both runqueue waiting and on-CPU work. |
| `exec_cpu_us`| `CLOCK_THREAD_CPUTIME_ID` | CPU time consumed by the busy loop. This clock advances **only while the thread is on a CPU**, so it excludes preemptions. By construction it should be ≈ C for every invocation. |
| `miss`       | derived           | `1` if `finish_ns > release_ns + period_ns`, else `0`. |

All threads inside one process are aligned to the same absolute start time
so contention begins simultaneously. The first 20 invocations of every
task are dropped as warm-up by `analyze.py`.

## 3. Deadline-miss definition

We use **implicit deadlines**: the deadline of invocation `i` equals the
release time of invocation `i+1`. Concretely:

```
deadline_i = release_i + period
miss_i     = (finish_i > deadline_i)
```

A miss therefore means *the job did not finish before the next job of the
same task became eligible to run.* The miss column is computed inline by
the workload (no post-processing). `miss_count` is the sum over all
non-warmup invocations and `miss_rate = miss_count / n`.

## 4. Variance / percentile metrics

For each task, `analyze.py` reports for both `exec_cpu_us` and
`response_us`:

- **mean** — arithmetic mean.
- **var**  — sample variance (`ddof=1`).
- **std**  — sample standard deviation.
- **p50, p95, p99, max** — percentiles using `numpy.percentile`.

The two metrics tell different stories:

- `exec_cpu_us` measures **work**. It should track the budgeted C ± a few
  µs regardless of contention, because `CLOCK_THREAD_CPUTIME_ID` does not
  advance while the thread is preempted.
- `response_us` measures **latency**. Under contention it grows with the
  amount of time the thread had to wait in the CFS runqueue. The gap
  `response_us − exec_cpu_us` is essentially **time spent ready but not
  running**.

## 5. How to read the results

A vanilla-CFS scheduler is **not** deadline-aware. With 6 CPU-bound
threads competing for 2 vCPUs, CFS rotates them in roughly equal time
slices (`sched_min_granularity` ~ a few ms). Two consequences:

1. **Short-period tasks miss most.** A task with `T = 20 ms` has at most
   one CFS rotation of slack; any extra preemption pushes the finish over
   the deadline.
2. **Variance of `response_us` is enormous compared to `exec_cpu_us`.**
   Several orders of magnitude difference is normal in this experiment;
   that gap *is* the scheduler-induced jitter.

In `summary.txt` you should observe:

- `exec_var` ≈ a few µs² → the workload itself is deterministic.
- `resp_var` ≈ 10⁸–10⁹ µs² → response time is dominated by waiting.
- Non-zero `miss_rate` on every task → CFS misses periodic deadlines even
  with 20 % CPU headroom.
- Roughly equal miss rates between `cA_*` and `cB_*` → no container is
  being starved; both are equally affected by the scheduler.

## 6. The CDF plot

`cdf.png` shows the empirical CDF of `response_us` for each task. The
deadline of a task is at `x = period_us`, so:

- The fraction of the curve at or to the **left** of `x = period_us` is
  the on-time rate.
- Everything to the **right** is missed deadlines.

If the curves for `cA_taskN` and `cB_taskN` overlap, the two containers
are equally hurt — pointing to the scheduler, not container-level
starvation, as the cause of the misses.

## 7. Files in this folder

| File | Origin | Content |
|---|---|---|
| `cA_task0.csv` … `cA_task2.csv` | container `rt-a` | Per-invocation timing for tasks 0/1/2 of container A. |
| `cB_task0.csv` … `cB_task2.csv` | container `rt-b` | Per-invocation timing for tasks 0/1/2 of container B. |
| `cA_proc_stat.csv`, `cB_proc_stat.csv` | per-container sampler | 1 Hz snapshots of `/proc/stat` cpu lines (steal/idle/system ticks) during the run. Useful as a sanity check on whether the host VM was being preempted by the hypervisor. |
| `summary.txt` | `analyze.py` | Aggregated per-task statistics (mean/variance/percentiles/miss rate). |
| `cdf.png`     | `analyze.py` | Empirical CDF of `response_us` per task. |

## 8. Reproducing

From the experiment root:

```bash
# (re-)build & push image (linux/arm64 for D-ps SKUs):
az acr build --registry <registry> --platform linux/arm64 --image rt-contended:0.1 .

kubectl apply -f k8s/workload-contended.yaml          # after substituting IMAGE
# wait until both containers print "[cX] done."
kubectl cp rt-contended:/out/<file> ./results/<run>/<file> -c rt-a   # for each CSV
kubectl delete pod rt-contended

python analyze.py results/<run>/
```
